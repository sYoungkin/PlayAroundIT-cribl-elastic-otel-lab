# Fleet Server Setup

Documentation for standing up **Fleet Server** on the Kibana node (`kibana-1`)
in the `PlayAroundIT-cribl-elastic-otel-lab` environment. Fleet Server is the
control plane that manages Elastic Agent enrollment, policies, and outputs.

---

## Overview

Fleet Server runs co-located on `kibana-1` and connects to Elasticsearch on
`elastic-1` over TLS with **full certificate validation** (consistent with how
Cribl connects to Elasticsearch). It listens on port **8220** for agent
enrollment and check-in.

| Property            | Value                              |
|---------------------|------------------------------------|
| Host node           | `kibana-1` (co-located with Kibana)|
| Fleet Server port   | `8220`                             |
| ES connection       | `https://elasticsearch:9200`       |
| ES cert validation  | Full (CA + hostname)               |
| CA used             | `/etc/kibana/certs/ca.crt`         |

---

## Why these settings (vs. the auto-generated command)

Elastic's Fleet UI generates an install command that is **not** correct for a
distributed, TLS-enabled lab. The generated command looked like:

```bash
sudo ./elastic-agent install \
  --fleet-server-es=http://localhost:9200 \
  --fleet-server-service-token=<token> \
  --fleet-server-policy=fleet-server-policy \
  --fleet-server-port=8220 \
  --install-servers
```

Three things were wrong for this environment:

1. **`localhost` â†’ `elasticsearch`.** The command runs on `kibana-1`, but
   Elasticsearch is on a *separate* VM (`elastic-1`). `localhost:9200` would look
   for ES on the Kibana box, where nothing listens. It must target `elastic-1`.

2. **`http://` â†’ `https://`.** Elasticsearch is TLS-only on its HTTP layer and
   rejects plain HTTP.

3. **Missing CA.** Because ES presents a lab-CA cert, Fleet Server must trust
   that CA â€” otherwise validation fails. Added via `--fleet-server-es-ca`.

### Hostname, not IP (the SAN/hostname rule again)

Same rule learned with Cribl: the ES server certificate is valid for the name
`elasticsearch`, **not** for an IP. Connecting by IP with validation on would
fail on hostname mismatch. So we connect by the name `elasticsearch` and resolve
it via `/etc/hosts`.

## Setup Steps (on `kibana-1`)

### 1. Resolve the `elasticsearch` hostname

Get `elastic-1`'s current DHCP IP (from the host):
```bash
vagrant ssh elastic-1 -c "hostname -I | awk '{print \$1}'"
```

Add it to `/etc/hosts` on `kibana-1`:
```bash
echo "<elastic-1-ip> elasticsearch" | sudo tee -a /etc/hosts
```

### 2. Confirm the CA is present

The CA was placed during Kibana provisioning:
```bash
ls -l /etc/kibana/certs/ca.crt
```
(Owned `root:kibana`, `640` â€” readable by the root-run installer.)

### 3. Install Fleet Server

```bash
sudo ./elastic-agent install \
  --fleet-server-es=https://elasticsearch:9200 \
  --fleet-server-service-token=<token> \
  --fleet-server-policy=fleet-server-policy \
  --fleet-server-port=8220 \
  --fleet-server-es-ca=/etc/kibana/certs/ca.crt \
  --install-servers
```

The `<token>` (service token) is generated by the Fleet UI when you set up Fleet
Server. The three corrections from the generated command: `https://`,
`elasticsearch` (not `localhost`), and the added `--fleet-server-es-ca` line.

> If the installer can't read the CA from `/etc/kibana/certs/`, copy it to a
> neutral path first (`sudo cp /etc/kibana/certs/ca.crt /tmp/ca.crt`) and point
> `--fleet-server-es-ca` there.

## Verification

```bash
sudo elastic-agent status
```
Expect a healthy/online status. In the Fleet UI (Kibana â†’ Fleet), the Fleet
Server should show as **Healthy**.
